## fetchの扱い
fetchをそのままService内で使うと以下の問題がある。
- fetchはグローバルオブジェクトである。テストの際にmockに置き換えられない。
  - 今回はテストしないからシンプルさの方が重視。
- WebAPIsが使えない環境では動作しない。
  - 使えないのは古いブラウザ（IEなど）や古いNode（17以前）みたいだから、そこまで気にしなくてもよさそう。

### 対処法
**① 使えない場合は例外を投げる。**
- 実行環境が変わる場合も対応できる（例外だが...）
```ts
const fetchFn =
	globalThis.fetch?.bind(globalThis);

if (!fetchFn) {
	throw new Error("fetch is unavailable");
}
```

**② DIでfetchメソッドを置き換え可能にする。**
- fetchメソッドをmockに置き換えて単体テスト実施できる。
- 使えない場合の対処は呼び出し元で行う。
```ts
// 呼び出し元 引数として渡すfetchメソッドを置き換え可能。
const api = createApiClient(globalThis.fetch);

// 呼び出し先
type FetchFn = typeof fetch;

export function createApiClient(fetchFn: FetchFn) {
	return {
		get: async (url: string) => {
			const res = await fetchFn(url);
			return res.json();
		},
	};
}
```

## クロージャを用いたファクトリの実装
- notion・databaseIdをcreateNotionService内に隠ぺいできる。
- createExpensePage、fetchExpenseCategoryRecordsに引数としてnotion・databaseIdを渡す必要がなくなる。
```ts
export function createNotionService(env: Bindings) {
	const notion = new Client({ auth: env.NOTION_API_KEY });
	const databaseId = env.NOTION_DATABASE_ID;

	return {
		createExpensePage(expense: ExpenseRequest) {
			return createExpensePage(notion, databaseId, expense);
		},
		fetchExpenseCategoryRecords() {
			return fetchExpenseCategoryRecords(notion, databaseId);
		},
	};
}
```
