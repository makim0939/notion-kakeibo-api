## Notion一本化したい理由

- SpreadSheetはスマホで扱いずらい。
- SpreadSheetは開くのがめんどい。
- iPhone → GAS → SpreadSheet → GAS → Notion は複雑。
    - iPhone → CloudFlareWorkers → Notion の方がシンプルだし書きやすい。
- データをSpreadSheetに持ちつつサマリをNotionで出すのが大変。

## API設計

### **POST `/expenses`**

支出を登録する。

**パラメータ**

| 項目 | 型 | 説明 |
| --- | --- | --- |
| name | string | 支出の名前 |
| amount | number | 金額 |
| paymentMethod | string | 支払い方法（カード／現金） |
| date | Date | 購入日（YYYY-MM-DD） |
| category | “カード” | ”現金” | null | カテゴリ。未指定の場合自動決定 |

**Response**

```json
// 成功
{
  "success": true,
  "pageId": "1f8c1d7d-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "category": "食費"
}

// 失敗
{
  "success": false,
  "message": "Invalid amount"
}
```

### バリデーション

| 項目  | 条件  | エラー内容 |
| --- | --- | --- |
| name | - | - |
| amount | 整数以外は禁止（負の整数は許容） | Invalid amount |
| paymentMethod  | “カード” | “現金” | null 以外は禁止 | Invalid payment method |
| date  |  YYYY-MM-DD形式以外は禁止 | Invalid date |
| date | 存在しない日付は禁止 | Invalid date |

### カテゴリ自動決定ロジック

支出名に含まれるキーワードをもとにカテゴリを決定する。

**適用条件**

- クライアントからのリクエストでcategoryが指定されていないこと。
- 指定されていた場合はその値をそのまま用いる（ロジックには通さない）

**キーワード管理**

- Cloudflare Workers内で定数として定義したキーワード + NotionDBから都度取得する過去の記録を用いる。

**判定ルール**

```tsx
const CATEGORY_KEYWORDS = {
	食費: ["昼ごはん", "ファミマ", "ローソン"],
	日用品: ["Amazon", "マツキヨ"],
	旅行: ["旅行", "ホテル"]
};
```

- 支出名にキーワードを含む場合、そのカテゴリを採用する。
- 最も当てはまるキーワードが多いカテゴリを採用する。
- 一致しない場合は「未分類」とする。

**認証**

- ChatGPTの指摘

        - `YYYY-MM-DD` なら日時ではなく日付。
    - ✅ `paymentMethod` と `method` が混在
        - APIパラメータ名を統一する。
    - ✅ `category` の扱いを明確化
        - 「未指定なら自動決定」なのか「常にクライアント指定」なのかを決める。
        - 現状、フロントでカテゴリ入力する記述と、Workersで自動決定する記述が混在。
    - ✅ カテゴリ自動決定ロジックの責務が曖昧
        - 「クライアントが送る」のか「Workersが決める」のかを整理。

        - 外部仕様では「カード固定」に近いが、APIでは自由入力になっている。
        - → クライアント側でカバーするため問題なし。APIはどのクライアントから受け取ったかを考慮しない。
    - ✅ `POST /expenses` のバリデーション不足
        - 金額0以下
        - 空文字
        - 不正日付
        - 未知の支払い方法
        をどう扱うか定義すると実装しやすい。
    - ✅ OCR失敗時の挙動が未定義
        - MVPで無視でもよいが、「金額取得失敗時は手入力にフォールバック」など最低限決めると安全。
        - → クライアント側でカバーするためAPI設計には定義しない。
    - ✅カテゴリキーワードをWorkers内定数にすると運用コストが高い
        - 将来的にNotionDBやJSON化したくなる可能性あり。
        - ただしMVPなら十分合理的。
    - ✅「支出名」の正規化方針がない
        - 例: `ファミマ`, `ファミリーマート`, `ﾌｧﾐﾏ`
        - カテゴリ精度に直結する。
        - → ひとまずユーザ任せで統一する。実現コストが高いと思う。
    - ✅ レスポンスが最小限すぎる
        - 作成したNotionページIDやカテゴリ結果を返すとデバッグしやすい。
    - ✅ Notion API失敗時のリトライ方針なし
        - 一時エラー時の扱いを軽く決めておくと良い。
        - → MVPではリトライは不要。実用で問題があれば検討
    - ✅ `NOTION_EXPENCES_DB_ID`
